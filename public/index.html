        Zero Trust Architecture

        Policy flow and enforcement logic

        Client

        User + Device

        Identity begins with credentials, OTP verification, and browser or device context.

        PEP

        Policy Enforcement Point

        Every access request is checked before users reach protected application surfaces.

        PDP

        Policy Decision Point

        Trust score, role, device posture, and simulation alerts determine the decision.

        Resource

        Protected Resource

        Admin, employee, and guest areas receive different visibility according to policy.

        Threat Model

        Assume breach and limit lateral movement

        Primary Assets

        Identity records, privileged access paths, audit history, and protected internal resources.

        Likely Attackers

        Credential thieves, malicious insiders, bots attempting brute force, and session hijackers.

        Entry Points

        Login forms, OTP workflow, role elevation requests, and browser session reuse.

        Control Strategy

        Continuous verification, least privilege, monitoring, event logging, and step-up responses.

        Demo Scope

        What this demo proves and what should come next

        Implemented here

        • Role-based access separation
        • OTP-based step-up verification
        • Device and browser trust scoring
        • Admin monitoring and attack simulation

        Next production upgrades

        • Server-side token validation and session storage
        • OAuth or SSO with MFA
        • Real audit logs, rate limiting, and anomaly detection
        • Database-backed RBAC or ABAC policy engine
